# AI 日报系统搭建思路（对外说明）v3.0.2

## 这套系统要解决什么问题

目标很简单：

1. 让我自己尽可能快地掌握当天最值得关注的 AI 资讯
2. 把真正值得看的内容，快速整理后分享给群友

这不是一个“做一份很完整的媒体日报”的系统，而是一个：

- 先抓全
- 再筛重点
- 再高效分享

的个人化 AI 资讯工作流。

## 为什么我会这样搭

我最终采用的是：

- 本地数据库
- Hermes 抓取和快报
- OpenClaw 做正式整理和分享输出
- 脚本优先，大模型只做判断和表达

核心原因只有一个：

**我需要的是“稳定、可信、可持续运行”的系统，而不是一次性看起来聪明的 demo。**

在信息系统里，只要涉及：

- 标题
- 链接
- 来源
- 时间
- 去重
- 状态流转

这些事实层内容，我都尽量放到脚本里处理，不交给大模型自由生成。

因为真正长期运行下来，最怕的不是系统“笨一点”，而是：

- 幻觉
- 状态错乱
- 重复发送
- 标题和链接对不上
- 看起来流畅，但信息不可信

## 整体架构

这套系统是双层结构：

### 1. Hermes：提醒层

Hermes 负责：

- 抓取 AI 资讯
- 写入本地 SQLite
- 做基础清洗、分类、打分
- 在固定时间点发送“这一轮最值得关注的信息”

Hermes 的作用不是写完整长文，而是让你快速知道：

**现在最值得我点开的东西是什么。**

### 2. OpenClaw：正式输出层

OpenClaw 负责：

- 从结构化候选里做正式选题
- 对少量入选条目读原文
- 输出更适合分享的内容

当前这套思路里，OpenClaw 重点不是做复杂大刊，而是做：

- 信息卡片
- 文本分析
- 更适合群分享的整理结果

这里我没有把它设计成依赖生图大模型的重型链路。

原因也很直接：

- 微信群分享更看重信息本身
- 个人快速掌握资讯更看重速度和可信度
- 不接入生图模型，链路更短、成本更低、稳定性更高

## 为什么一定要有本地数据库

我选择本地 SQLite 作为唯一主库，不把飞书多维表当主库。

原因：

1. 本地数据库是事实层的唯一 source of truth
2. 抓取、去重、状态写回、后续补跑都更稳定
3. 不依赖第三方平台的字段限制和接口波动
4. 观察面、快报、正式日报都可以复用同一份结构化素材池

飞书多维表在这套系统里的角色只是：

- 观察面
- 可视化同步视图

不是主链路数据库。

## 为什么要“脚本优先，大模型靠后”

这是这套系统里最重要的一条原则。

我最后确认下来的做法是：

### 脚本负责事实层

- 抓取原标题
- 保存原文链接
- 保存来源和时间
- 去重
- 候选池构建
- 状态回写
- 观察面同步

### 大模型负责判断层

- 哪些内容更值得关注
- 哪些应该入选
- 怎么写摘要
- 怎么写分析

这样做的好处是：

1. 事实可信
2. 系统可复现
3. 可排查
4. 长期运行更稳

一句话就是：

**能脚本化的事情，尽量不要先交给大模型。**

## 为什么这套系统适合微信群分享和个人快速掌握资讯

因为它的目标不是“写一份很漂亮的大报告”，而是解决两个非常实际的问题：

### 对个人

你每天不需要自己刷十几个来源。

系统会先帮你把信息抓回来，再告诉你：

- 哪些是最新的
- 哪些值得看
- 哪些只是噪音

### 对微信群

群分享最适合的不是冗长长文，而是：

- 一眼能看懂的信息卡片
- 一段明确的摘要
- 一段为什么值得关注的分析

这套结构天然适合做：

- 轻量日报
- 群内资讯播报
- 小范围高质量分享

## 当前状态：适合个人可用

目前这套系统的定位，我会定义为：

**个人可用，且可以小范围稳定分享。**

这意味着：

- 自己日常掌握 AI 资讯，已经足够用了
- 分享给群友，也已经有实际价值
- 但如果要做成“媒体级产品”，还需要继续增强模板、分发和稳定性

## 这套思路为什么容易扩展到别的行业

这套架构不是只适合 AI。

它本质上是一个通用的信息流系统：

1. 指定一组信源
2. 用脚本抓取事实
3. 用本地数据库做结构化沉淀
4. 用规则和模型做轻量筛选
5. 用一个提醒层 + 一个正式输出层来分发

所以如果换成别的领域，比如：

- 跨境电商
- 游戏行业
- 智能硬件
- 机器人
- 证券/投研
- 本地生活

理论上都可以复用同样的架构。

真正需要调整的不是主架构，而是：

- 信源列表
- 分类规则
- 评分逻辑
- 最终输出模板

也就是说，它天然适合做：

**面向不同垂直行业的信息系统个性化配置。**

## 如果交给别的 agent 适配，应该怎么做

我的建议是：

### 先不要让 agent 从头自由发挥

而是把这套思路明确给它：

1. 用本地数据库做主库
2. 抓取层脚本化
3. 标题和链接必须保真
4. 先做提醒层，再做正式输出层
5. 大模型只做筛选、摘要、分析

然后让 agent 基于具体行业再做：

- 信源替换
- 字段调整
- 分类适配
- 输出样式适配

这样收敛最快，也最稳。

## 部署建议

如果要部署这套系统，我建议：

### 优先给有 coding plan / token 预算的人用

因为首次搭建这类系统时，最消耗 token 的不是日常运行，而是：

- 架构收敛
- prompt 调整
- 分类规则校准
- 故障排查
- 生成格式反复验证

也就是说：

**第一次把系统搭起来，通常是最费 token 的阶段。**

反而当结构稳定后，日常运行会更轻。

所以如果要让别人复刻，我的建议是：

1. 最好有 coding plan
2. 最好按现成架构部署，不要从零自由设计
3. 尽量把事实层先脚本化，再逐步引入模型

## 一句话总结

这套系统的搭建思路不是“让大模型自动写日报”，而是：

**用本地数据库锁住事实，用 Hermes 保证信息同频，用脚本做筛选和约束，用 OpenClaw 输出更适合分享的卡片和分析。**

它现在最适合的场景是：

- 个人快速掌握 AI 资讯
- 微信群内分享值得重点关注的当日信息
- 以及后续面向其他垂直行业做个性化适配
